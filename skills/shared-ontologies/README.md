# Shared Ontologies

An evidence-based skill for designing, evolving, and governing shared ontologies in software systems. Distilled from 50+ peer-reviewed or canonical sources spanning ontology engineering (Gruber, Grüninger & Fox, METHONTOLOGY, NeOn), semantic web (OWL, OBDA, SHACL/ShEx), software-engineering ontology (Calero, Ruiz & Piattini; Happel & Seedorf), biomedical ontology (OBO Foundry, Gene Ontology, MIREOT), data integration (Wache, Xiao, Corcho), schema and ontology matching (Rahm & Bernstein, Euzenat & Shvaiko), and ontology evolution (Klein & Fensel, Noy & Musen, Flouris, Zablith, PAV).

## Scope and Factuality Framing

**Scope.** Shared ontologies in software/data systems — semantic artifacts used across services, schemas, APIs, knowledge graphs, validation layers, annotations, data-access mappings, or architecture artifacts.

**Factuality controls.** Each reference separates source-backed evidence (synthesized from cited literature) from derived engineering guidance. Per-reference frontmatter records `evidenceStrength` and `corroborationCount` in place of subjective confidence levels. Code examples are synthetic implementation sketches that illustrate the derived engineering rule; they are not examples observed in, or claimed by, the cited papers.

## Structure

- `SKILL.md` — generated: spec entrypoint with TOC linking into `references/`
- `AGENTS.md` — generated: long-form fallback for the AGENTS.md convention
- `metadata.json` — skill-level metadata (name, description, license, body, citations)
- `references/` — source references grouped by section prefix (`requirements-`, `architecture-`, `validation-`, `evolution-`, `governance-`); each file is loaded by the agent on demand

## When to Reach for It

- Standing up a shared vocabulary that more than one service is going to commit to
- Integrating data across systems that historically used distinct local schemas
- Picking between RDF/SKOS, an OWL 2 profile, OWL DL, rule languages, and SHACL
- Gating an ontology release before downstream consumers depend on it
- Versioning a semantic change that affects published data, queries, or generated code
- Aligning a new local schema with an existing reference or upper ontology
- Reviewing a proposal that adds, renames, deprecates, or splits ontology terms

## Getting Started

1. Read `SKILL.md` for the reference index.
2. Open the relevant files in `references/` for source-backed evidence, derived engineering guidance, and synthetic implementation sketches.
3. For a single-file view of every reference, read `AGENTS.md`.

## Authoring

- Each reference file is `references/<prefix>-<slug>.md` with frontmatter, source-backed-evidence + derived-engineering-guidance body, and synthetic implementation sketches.
- Section ordering and impact live in `references/_sections.md`.
- `SKILL.md` and `AGENTS.md` are regenerated by `packages/skills-build`. Do not hand-edit them.
