# Refactoring Playbook

An evidence-based skill for planning, executing, and reviewing refactoring work in any code base. Distilled from empirical software engineering research (Opdyke, Fowler, Mens & Tourwé, large-scale industrial studies at Microsoft and Google, plus recent LLM-assisted refactoring evaluations).

## Structure

- `SKILL.md` — generated: spec entrypoint with TOC linking into `references/`
- `AGENTS.md` — generated: long-form fallback for the AGENTS.md convention
- `metadata.json` — skill-level metadata (name, description, license, body, citations)
- `references/` — source references grouped by section prefix (`triggers-`, `safety-`, `review-`, `automation-`, `measure-`); each file is loaded by the agent on demand
- `scripts/` — bundled helpers (`hotspots.sh` for top-decile churn ranking)

## When to Reach for It

- Deciding whether a refactor is worth doing now or should be deferred
- Choosing between a single large refactor PR and a sequence of small ones
- Scoping a framework or API migration
- Reviewing a PR that mixes refactoring with feature or bug work
- Defining success criteria before declaring a refactor complete

## Getting Started

1. Read `SKILL.md` for the reference index.
2. Open the relevant files in `references/` for bad/good examples.
3. For a single-file view of every reference, read `AGENTS.md`.

## Authoring

- Each reference file is `references/<prefix>-<slug>.md` with frontmatter, explanation, and bad+good examples.
- Section ordering and impact live in `references/_sections.md`.
- `SKILL.md` and `AGENTS.md` are regenerated by `packages/skills-build`. Do not hand-edit them.
