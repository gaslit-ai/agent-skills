# Refactoring Playbook

An evidence-based skill for planning, executing, and reviewing refactoring work in any code base. Distilled from empirical software engineering research (Opdyke, Fowler, Mens & Tourwé, large-scale industrial studies at Microsoft and Google, plus recent LLM-assisted refactoring evaluations).

## Structure

- `SKILL.md` — runtime trigger, applicability, and quick reference
- `metadata.json` — version and abstract metadata
- `rules/` — source rules grouped by section prefix (`triggers-`, `safety-`, `review-`, `automation-`, `measure-`)
- `AGENTS.md` — compiled long-form guide (generated by the build package)

## When to Reach for It

- Deciding whether a refactor is worth doing now or should be deferred
- Choosing between a single large refactor PR and a sequence of small ones
- Scoping a framework or API migration
- Reviewing a PR that mixes refactoring with feature or bug work
- Defining success criteria before declaring a refactor complete

## Getting Started

1. Read `SKILL.md` for triggers and the rule index.
2. Open the relevant rule files in `rules/` for bad/good examples.
3. For a printable single-file view, read `AGENTS.md`.

## Authoring

- Each rule file is `<prefix>-<slug>.md` with frontmatter, explanation, and bad+good examples.
- Section ordering and impact live in `rules/_sections.md`.
- `AGENTS.md` is regenerated by `packages/refactoring-playbook-build`. Do not hand-edit it.
